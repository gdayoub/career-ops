## F) STAR+R Interview Stories

**1. Tell me about building something with a non-trivial rendering challenge (Domain relevance)**
- **S:** CoverMe needed real-time LaTeX document preview in the browser — no library handled my 3-template system
- **T:** Build a renderer that was instant, layout-correct, and maintainable
- **A:** Built a shared parser that extracted LaTeX document structure (sections, environments, commands) and mapped each node type to CSS-styled HTML; zero server roundtrip using client-side rendering
- **R:** Real-time preview working at launch; 3 layouts all handled by one parsing pass
- **R:** Similar to what Figma does with canvas — parse a structured representation, render it correctly in the browser

**2. Tell me about a time you designed a system with constraints (Technical design)**
- **S:** LionPlanner needed to generate schedules efficiently — exponential search space with multiple constraints
- **T:** Design an algorithm that's fast enough for real-time use
- **A:** Backtracking with early pruning (time conflict detection at each level), multi-criteria scoring (professor rating × GPA safety × workload balance), constraint ordering to prune the deepest branches first
- **R:** <500ms for typical Penn State inputs; 200 valid schedules per query
- **R:** Would add memoization of partial solutions for larger catalogs

**3. Tell me about shipping a user-facing feature iteratively**
- **S:** CoverMe users reported that generated letters felt "too formal"
- **T:** Add tone control without rebuilding the generation pipeline
- **A:** Added tone slider (formal → casual) to the prompt context; wrapped in iterative refinement loop where users could re-generate with feedback
- **R:** Session depth increased — users iterate 3–5 times per letter; tone is now the #1 feature request mentioned in feedback
- **R:** User feedback loops are faster to implement than I thought — a prompt parameter change had UX impact

**4. Tell me about a time you collaborated in a cross-functional environment**
- **S:** At Seismos, I needed engineering team sign-off on my pipeline outputs before they could go to clients
- **T:** Build trust between data science outputs and QC engineers
- **A:** Created a validation checklist (field types, unit assertions, known-value spot checks) that engineers could run themselves; documented failure modes and how to distinguish extraction errors from genuine data anomalies
- **R:** Engineers approved outputs on first review in all subsequent cycles; built a shared vocabulary for data quality
- **R:** Documentation of failure modes proved more valuable than the pipeline itself

**5. How would you improve Figma? (Product intuition)**
- **S:** FigJam collaboration often has awkward version merging when two people edit the same sticky note cluster
- **T:** Reduce friction in real-time collaborative editing
- **A (hypothetical):** Add operational transforms (OT) or CRDT-based sticky-note merging with visual diff highlighting — show exactly what changed when your collaborator's edit arrives
- **R:** Reduces "who moved my sticky?" confusion in distributed team retrospectives
- **R:** Demonstrates understanding of Figma's core collaboration infrastructure challenge
