## F) STAR+R Interview Stories

**1. Tell me about your most technically complex project (NLP/AI relevance)**
- **S:** I needed to classify Arabic-language geopolitical news from 3 live sources with no existing dataset
- **T:** Build a robust NLP pipeline and classifier that runs in production
- **A:** Built Arabic preprocessing (Unicode normalization, stopword removal, camel-tools morphological analysis), extracted TF-IDF features, trained logistic regression with cross-validation, containerized with Docker Compose
- **R:** Live production classifier with Streamlit dashboard; achieves reliable topic categorization on novel articles
- **R:** Would fine-tune a multilingual BERT for better precision on ambiguous cases

**2. Tell me about owning a feature end-to-end (Product ownership)**
- **S:** CoverMe needed a real-time LaTeX document renderer with no off-the-shelf solution
- **T:** Build something that works instantly in the browser across 3 distinct template layouts
- **A:** Designed a shared parser architecture extracting LaTeX structure → HTML/CSS; zero server roundtrip; handled edge cases iteratively
- **R:** Real-time preview at launch across all templates; shared architecture saved ~30% code
- **R:** Would add progressive rendering for large documents

**3. Tell me about a time you shipped in a fast-paced environment (Speed + quality)**
- **S:** CoverMe needed full Stripe integration + subscription gating + letter history in the same sprint
- **T:** Don't break existing features while adding payment-gated access
- **A:** Staged the rollout: payment webhook first → access control second → letter history last; tested each stage independently
- **R:** Shipped without regressions; Stripe subscription is the primary revenue mechanism
- **R:** Learned to decompose risky changes into safe, staged deployments

**4. Tell me about collaborating with non-engineers (Communication)**
- **S:** At Seismos, my PDF extraction pipeline outputs were used directly by field engineers — not data scientists
- **T:** Make the outputs readable and trustworthy for a non-technical audience
- **A:** Changed output format from raw DataFrame to formatted Excel with labeled columns, units, and conditional highlighting; documented the pipeline with plain-English runbook
- **R:** Engineers adopted the tool without needing training; zero errors in client-facing deliverables
- **R:** Good output format is as important as correct extraction logic

**5. How would you improve a Grammarly feature? (Product intuition)**
- **S:** GrammarlyGO sometimes generates suggestions that sound like "AI" rather than the user's authentic voice
- **T:** Improve personalization fidelity
- **A (hypothetical):** Build a user-writing-style embedding trained on their historical documents; condition generation on style vector at inference time
- **R:** Suggestions would match the user's vocabulary, sentence length patterns, and formality level
- **R:** Shows I understand the gap between "good writing" and "writing that sounds like me"
