## F) STAR+R Interview Stories

**1. End-to-End Ownership: Describe a project you took from concept to production**
- **S:** CoverMe — I identified that job seekers had no tool to generate personalized, ATS-optimized cover letters quickly
- **T:** Build a complete, paid product solo — no team, no PM, no DevOps
- **A:** Designed the architecture (Next.js + Supabase + Stripe + Claude API), implemented 5 LaTeX templates with a custom real-time HTML renderer, integrated Stripe subscriptions with tiered access, deployed to Vercel with Row Level Security
- **R:** Live SaaS with paying users, multiple templates, production auth and billing working day one
- **R (Reflect):** Would have built a waitlist earlier to validate willingness to pay before full implementation

**2. Algorithmic Thinking: Describe a challenging technical problem you solved**
- **S:** LionPlanner needed to generate conflict-free Penn State class schedules — the input space was millions of combinations
- **T:** Generate up to 200 valid schedules in real time while satisfying multiple constraints (time conflicts, professor quality, GPA safety, workload balance)
- **A:** Designed a backtracking constraint solver that prunes invalid branches early; structured constraints by priority so the most restrictive filters run first
- **R:** Real-time generation on GCP Cloud Run — handles concurrent requests without timeout; 200 schedules returned in under 2 seconds
- **R:** Would add memoization for repeated subproblems to cut computation further

**3. Systems Thinking: Describe designing a system for reliability**
- **S:** The Arabic OSINT platform needed to ingest news from 3 sources continuously without duplicates or data loss
- **T:** Build a pipeline that deduplicates, normalizes Arabic text, classifies topics, and scores geopolitical threat — reliably and repeatably
- **A:** Built a PostgreSQL schema with hash-based deduplication; added Unicode normalization + camel-tools for Arabic preprocessing; containerized with Docker Compose so the whole stack restarts cleanly
- **R:** Live pipeline running with real-time threat scoring visible on Streamlit dashboard; container restarts without data loss
- **R:** Would add a dead-letter queue for failed articles and a retry mechanism for transient scraping failures

**4. Bias for Action: Describe moving fast despite uncertainty**
- **S:** At Seismos, PDF formats from field engineers varied wildly — no two PJRs looked the same
- **T:** Automate extraction within the internship window despite the format inconsistency being unknown at start
- **A:** Shipped an initial extractor for the most common format first; cataloged edge cases as they appeared; added regex-based fallbacks incrementally
- **R:** Covered 90%+ of PDFs with the primary parser; fallbacks caught the rest. Engineering teams got reports in seconds vs. hours
- **R:** Would have collected a sample of PDFs before writing any code to map the full variance space upfront
