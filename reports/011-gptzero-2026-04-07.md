## F) STAR+R Interview Stories

**1. Tell me about your NLP pipeline (Domain directly relevant to GPTZero)**
- **S:** Needed to classify Arabic geopolitical news into categories with no pre-existing dataset
- **T:** Build a pipeline that's accurate, reliable, and runs in production
- **A:** Built Arabic preprocessing (Unicode normalization, camel-tools, stopwords); TF-IDF feature extraction; logistic regression with 5-fold cross-validation; deployed with Streamlit + Docker Compose
- **R:** Live production classifier; achieves reliable topic categorization
- **R:** GPTZero's core problem (detecting AI-generated text) is also an NLP classification task — the pipeline architecture is directly analogous

**2. Tell me about working with LLMs in a product context**
- **S:** CoverMe's letter quality depended entirely on how well I engineered the Claude API prompt
- **T:** Make generated letters feel personalized, not generic
- **A:** Structured the prompt context with: job description key requirements, resume proof points mapped to requirements, tone preference, ATS keyword targets; added iterative refinement loop
- **R:** Users iterate 3–5 times per session — high engagement with the generation quality
- **R:** LLM prompting is engineering, not magic — the same principles (structured input, clear output format, explicit constraints) apply to fine-tuning

**3. Tell me about a production system you maintain**
- **S:** OSINT platform runs continuously — ingesting and classifying Arabic news
- **T:** Keep it running without babysitting
- **A:** Docker Compose with restart policies; PostgreSQL backups; deduplication prevents database bloat; Streamlit health check on startup; manual alert on ingestion failure
- **R:** Zero unplanned downtime since deployment
- **R:** Would add Prometheus metrics + Grafana dashboards for a team environment

**4. Tell me about learning from a failure**
- **S:** CoverMe's first LaTeX renderer broke on edge cases — unusual characters, nested environments, special commands
- **T:** Fix without rewriting the entire renderer
- **A:** Added a test suite with real-world LaTeX samples; identified 15 edge cases; wrote targeted fixes; added regression tests for each
- **R:** Renderer handles all real-world template inputs; test suite catches regressions
- **R:** Test-driven development is worth the upfront investment, especially for parsers

**5. Why GPTZero? (Motivation)**
- AI-generated text detection is one of the most important applied NLP problems of this decade — it's about the integrity of information, not just a product feature
- GPTZero's approach (combining perplexity, burstiness, and ML detection) is technically deep — I want to work on the models, not just the API layer
- The policy impact (cited by state-level policymakers, NYT) means the work has stakes beyond user metrics
