## F) STAR+R Interview Stories

**1. Tell me about a full-stack feature you built top-to-bottom**
- **S:** CoverMe needed tiered access — free users get 3 letters, paid users get unlimited
- **T:** Build access control that worked correctly across all entry points without performance hit
- **A:** Stripe subscription webhook → updates Supabase user record → Supabase Row Level Security enforces at DB layer → frontend reads tier from auth context → API validates before generation
- **R:** Access control works at every layer; billing edge cases (failed payments, upgrades mid-session) handled correctly
- **R:** Learned that auth-layer enforcement is safer than application-layer checks

**2. Tell me about a data pipeline you designed (Relevant to Klaviyo's core)**
- **S:** Arabic OSINT platform needed to ingest from 3 live news APIs without duplicates or data loss
- **T:** Design a reliable ingestion pipeline with deduplication and normalization
- **A:** URL + title hashing for dedup; Unicode normalization for Arabic encoding variants; structured storage in PostgreSQL with timestamps, source tags, and classification scores
- **R:** Zero duplicate articles in production; pipeline runs continuously, ingesting and classifying 100+ articles/day
- **R:** Would add a dead-letter queue for failed ingestion events — first thing I'd add at Klaviyo scale

**3. Tell me about handling a complex data structure problem (CS fundamentals)**
- **S:** LionPlanner had to handle schedule generation with complex constraint relationships
- **T:** Represent the constraint graph efficiently and traverse it without exponential blowup
- **A:** Backtracking with early pruning — represent each time slot as a bitmask, detect conflicts in O(1) per check; order constraints by most-constrained-variable first
- **R:** Generated 200 valid schedules in <500ms for typical input size
- **R:** Would explore CSP solvers (Google OR-Tools) for more complex constraint sets

**4. Tell me about selecting the right tool for a problem**
- **S:** LionPlanner needed a database that could handle structured course data and be deployed on GCP cheaply
- **T:** Choose between SQLite (simple), PostgreSQL (powerful), Firestore (NoSQL managed), or Cloud Spanner (expensive)
- **A:** Evaluated each: SQLite — no concurrent access; Firestore — wrong data model for relational course constraints; Cloud Spanner — overkill cost; chose Cloud SQL PostgreSQL for relational integrity + GCP integration + Prisma ORM support
- **R:** Right call — schema migrations and relational queries worked cleanly throughout development
- **R:** The tool selection process taught me to evaluate against specific use-case requirements, not just popularity

**5. Why Klaviyo? (Motivation)**
- CoverMe gave me a direct look at how marketing automation data flows — Stripe events, subscription state, user engagement signals all need to be processed reliably and acted on in near-real-time
- Klaviyo is the best engineering implementation of this problem at scale: billions of events, 150K+ customers, petabyte data
- The internship's scope (petabyte-scale pipelines at age 19) is a rare opportunity to build experience most engineers don't get until mid-career
