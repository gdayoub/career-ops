## F) STAR+R Interview Stories

**1. Tell me about a system you built that needed to be reliable (Infra relevance)**
- **S:** CoverMe's Stripe webhooks needed to be processed exactly-once — duplicate events would double-charge users
- **T:** Build idempotent webhook processing
- **A:** Used Stripe's idempotency key in the webhook handler; stored processed event IDs in Supabase; rejected duplicate events before triggering subscription updates
- **R:** Zero double-charge incidents since launch
- **R:** Idempotency as a design principle applies to any distributed system — including Cloudflare's request pipeline

**2. Tell me about working with distributed/containerized infrastructure**
- **S:** Arabic OSINT platform needed 3 co-dependent services (PostgreSQL, NLP worker, Streamlit dashboard) running reliably together
- **T:** Orchestrate them without managing individual process lifecycles
- **A:** Docker Compose with health checks (Postgres healthcheck → NLP worker starts → dashboard starts); environment-specific .env files; named volumes for data persistence
- **R:** One `docker compose up` command starts the entire stack; portable across dev/prod environments
- **R:** Would add a reverse proxy (Nginx/Traefik) to expose services cleanly if deploying publicly

**3. Tell me about a time you learned a new technology quickly to solve a problem**
- **S:** LionPlanner needed GCP Cloud Run deployment — I had never used GCP before
- **T:** Deploy a containerized Next.js + PostgreSQL app to Cloud Run in a week
- **A:** Read GCP Cloud Run docs; containerized the app with a multi-stage Dockerfile; set up Cloud SQL, Secret Manager, and Cloud Build; troubleshot IAM permission issues iteratively
- **R:** Fully deployed production system on GCP in under a week
- **R:** GCP's IAM complexity was the biggest friction point — would use infrastructure-as-code (Terraform) to make permissions reproducible

**4. How would you improve Cloudflare Workers? (Product intuition)**
- **S:** Current Workers dev experience requires `wrangler` CLI context-switching — local/preview/production parity is sometimes unclear
- **T:** Reduce local-to-production surprises
- **A (hypothetical):** Add a local "shadow mode" that replays production request traffic against the local Worker instance, highlighting behavior differences
- **R:** Faster debugging cycles; fewer "works locally, fails in production" incidents
- **R:** Shows understanding of the key pain point in edge-compute development

**5. Why Cloudflare? (Motivation)**
- Cloudflare handles the internet's infrastructure at a scale that's hard to comprehend — the engineering challenges (millisecond routing, global consistency, DDoS mitigation) are unique
- The Workers platform is exactly the right abstraction layer for the next generation of distributed apps — serverless at the edge, close to users
- I've built on serverless (GCP Cloud Run, Vercel) and want to understand the infrastructure that makes it possible, not just consume it
